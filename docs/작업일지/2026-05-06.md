## 셀 4 Stage 5 pending 버그 수정

### 문제
Colab에서 셀 4(Stage 5) 실행 시 `quality_scores.json`이 일부 생성된 후 스크립트가 무한 pending.

### 원인
`ThreadPoolExecutor(SEED_THREADS=4)` 내부에서 `run_quality_scoring(workers=-1)` 호출
→ 각 스레드가 `ProcessPoolExecutor(N)` 추가 생성 (중첩 병렬)
→ Linux `fork` 시 부모 프로세스의 다른 스레드가 보유한 C 레벨 락(OpenCV 등)이
  자식 프로세스에 잠긴 채 복사됨 → deadlock.

일부 `quality_scores.json`이 생성되는 이유:
ProcessPoolExecutor 진입 전(소량 이미지 seed)은 순차 처리로 완료,
나머지 seed에서 fork deadlock으로 영구 pending.

### 수정

**`docs/작업일지/phase2_execute_oneclick.md`**
- 셀 1: `IMG_WORKERS = -1` → `IMG_WORKERS = 0` (이미지 단위 순차 처리)
- 셀 4: `_run_s5` 함수 인자로 `_stage4=drive_stage4` 추가 (클로저 명시적 값 캡처)
- 셀 4: progress 출력에 처리 seed 수 추가

### 동작 변경
- 이전: `ThreadPoolExecutor(4)` × `ProcessPoolExecutor(N)` = 중첩 병렬 → deadlock
- 이후: `ThreadPoolExecutor(4)` × 순차 이미지 처리 = 단일 레벨 병렬 → 안전

### 수행 카테고리 (✅ 적합 3개)

| 카테고리 | baseline | aroma_diffusion | aroma_ratio_10 | Δ(base→diff) |
|---|---|---|---|---|
| cashew | 0.888 | 0.879 | 0.728 | -0.009 |
| candle | 0.751 | 0.724 | 0.719 | -0.027 |
| capsules | 0.685 | **0.420** | **0.412** | **-0.265** |

### 관찰된 패턴

1. **전 카테고리 `baseline > aroma_diffusion`**: 합성 증강이 성능 개선 없이 일관되게 손상.

2. **합성 비율과 성능 역관계**:
   - `aroma_diffusion` (≈200% 합성): baseline 대비 소폭 손상
   - `aroma_ratio_10` (≈11% 합성): 더 큰 손상
   - 합성 풀이 클수록 pruning 후 품질이 높아져 상대적으로 나은 결과

3. **capsules 이상치**: AUROC 0.420 < 0.5 → score 방향 역전.
   합성 결함과 실제 결함 분포 간 도메인 갭이 극단적으로 큼.
   `image_best_f1`(0.647) ≈ aroma_ratio_10(0.647) — 동일한 합성 풀 사용 확인.

4. **baseline 강도와 손상 폭 상관**:
   - baseline 강할수록(cashew 0.888) → 손상 소폭
   - baseline 약할수록(capsules 0.685) → 손상 폭발적
   pretrained 피처가 약한 카테고리일수록 합성 품질에 민감.

## Phase 2 성능 개선 포인트 분석

### 현황 진단

현재 파이프라인이 성능을 개선하지 못하는 근본 원인:

```
합성 결함 분포 ≠ 실제 결함 분포
→ 분류기가 "합성 결함 스타일"을 학습
→ 실제 결함에 일반화 실패
→ baseline(pretrained 피처 거리) 대비 열위
```

### 개선 포인트

#### P1. pruning 기준 강화 (즉시 적용 가능)

**현재**: `pruning_ratio=0.5` → 상위 50% 선택
**문제**: 합성 풀이 작을 때 50% 안에도 저품질 포함

```yaml
# benchmark_experiment_phase2.yaml
dataset:
  pruning_ratio: 0.3  # 0.5 → 0.3: 상위 30%만 선택
```

기대 효과: 더 엄격한 품질 컷 → 합성·실제 결함 갭 축소

**주의**: 사용 가능한 합성 이미지 수 감소 → 합성 풀이 충분히 큰 카테고리에만 유효.
`셀 6-1` (Stage 4 합성 수 진단)으로 풀 크기 사전 확인 필요.

---

#### P2. 합성 풀 확대 (Stage 4 재실행 필요)

**현재**: `seed_ratio=0.25` (전체 seed의 25%), `max_images_per_seed=5`
**문제**: 풀이 작아 pruning 효과 제한

```yaml
synthesis:
  seed_ratio: 0.5          # 0.25 → 0.5
  max_images_per_seed: 10  # 5 → 10
```

기대 효과: pruning 전 풀 2~4배 확대 → 상위 품질 이미지 선택 폭 증가

**비용**: Stage 4 Diffusion 추론 시간 2~4배 증가.

---

#### P3. quality score 가중치 재조정

**현재**: `artifact:blur = 0.5:0.5`
**문제**: 합성 결함의 핵심 품질 지표가 blur인지 artifact인지 불명확

옵션:
```yaml
# blur 중심 (선명도 우선)
w_artifact: 0.3
w_blur: 0.7

# artifact 중심 (경계 자연스러움 우선)
w_artifact: 0.7
w_blur: 0.3
```

검증 방법: 각 가중치로 quality_scores.json 재계산(셀 4-0→4) 후 pruning 결과 시각적 확인.

---

#### P4. augmentation_ratio_by_domain 카테고리별 조정

**현재**: VisA 전체 `full: 2.0` 고정
**문제**: capsules처럼 합성 품질이 낮은 카테고리에 동일 비율 적용

접근: baseline AUROC가 낮은 카테고리(< 0.75)는 합성 비율 낮추거나 제외.
성능 손상이 확인된 카테고리를 벤치마크 분석에서 별도 취급.

---

#### P5. ControlNet 파인튜닝 (중장기)

**현재**: pretrained `lllyasviel/sd-controlnet-canny` 사용
**문제**: 도메인 특화 결함 텍스처 재현 한계

카테고리별 실제 결함 이미지로 ControlNet 파인튜닝 → 합성·실제 결함 분포 갭 축소.
가장 효과 클 것으로 예상되나 데이터·시간 비용 큼.
