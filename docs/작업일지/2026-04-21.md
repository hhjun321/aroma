## Phase 2 Diffusion ControlNet 입력 설계 수정

### 문제 (분석)

`stage4_diffusion_synthesis.py`의 `_synthesize_one()`에서
ControlNet control image를 `placements[0]["defect_path"]`,
즉 **Stage 2 변형 이미지(elastic warp)**에서 Canny 엣지로 추출하고 있었다.

이는 두 가지 이유로 잘못된 설계다.

1. **CASDA 검증 기반 불일치**: CASDA의 파라미터(`strength=0.7`, `guidance_scale=7.5`,
   `conditioning_scale=0.7`)는 **실제 결함 이미지**를 ControlNet 입력으로 실험하여 검증한 값이다.
   변형 이미지를 입력으로 사용하면 이 파라미터들이 더 이상 최적이 아닐 수 있다.

2. **연구 ablation 순수성 훼손**: Phase 2의 핵심 질문은 "합성 방식을 MPB → Diffusion으로
   바꾸면 품질이 개선되는가?"이다. Diffusion 경로에 MPB 고유 요소(Stage 2 elastic warp)가
   개입되면 개선 효과를 Diffusion에 귀속할 수 없다.

### 수정 내용

**`stage4_diffusion_synthesis.py`**
- `_synthesize_one()` 내 ControlNet control image 추출 소스 변경
  - 변경 전: `placements[0]["defect_path"]` (Stage 2 변형 이미지)
  - 변경 후: `seed_profile["seed_path"]` (Stage 1b 원본 seed 이미지)
- 모듈 docstring에 설계 원칙 명시 (CASDA 방식 준수, Stage 2 사용 범위 한정)

**`docs/작업일지/phase2_execute.md`**
- "Stage 0–3: Phase 1 결과 재사용" 섹션에 Stage 2/3 재사용 범위 표 추가
- Stage 2: ROI 위치·크기 계산에만 사용 (ControlNet 입력 아님)
- Stage 1b seed_profile["seed_path"]: ControlNet 입력 소스로 명시
- CASDA 방식 준수 설계 원칙 및 Drive 경로 주의사항 문서화

### Stage 2 역할 정리 (변경 후)

| 용도 | Phase 1 (MPB) | Phase 2 (Diffusion) |
|------|--------------|---------------------|
| ROI 위치·크기 | placement_map 통해 사용 ✓ | placement_map 통해 사용 ✓ |
| 합성 픽셀 소스 | Poisson blend 입력 ✓ | 사용 안 함 |
| ControlNet 입력 | 해당 없음 | **Stage 1b 원본 seed** ✓ |

## Stage 4 Diffusion 합성 성능 개선

### 문제 (분석)

`_synthesize_one()`이 **이미지마다** 다음 두 작업을 수행하고 있었다:
1. `seed_profile["seed_path"]` Drive 읽기 (동일 파일을 M번 반복)
2. `cv2.Canny()` 계산 (같은 결과를 M번 반복)
3. `PromptGenerator()` 인스턴스 생성 (M번 반복)

하나의 seed에 M개의 배경 이미지가 있을 때, 위 연산이 M회 중복 수행되었다.

### 수정 내용

**`stage4_diffusion_synthesis.py`**

- `_synthesize_one()` 시그니처 변경: `seed_profile` 대신 `control_pil`, `prompt_gen` 직접 수신
- `run_synthesis()`: 이미지 루프 진입 전 `control_pil`, `prompt_gen` 1회 계산
- `run_synthesis_batch()`: seed 루프 내, 이미지 루프 진입 전 `control_pil`, `prompt_gen` 1회 계산

**효과:** seed당 Drive 읽기 M→1회, Canny 계산 M→1회로 감소.

**`docs/작업일지/phase2_execute.md`**

Stage 4 Colab 셀에 로컬 SSD 캐시 패턴 추가:
- `LOCAL_TMP = /content/tmp_stage4`에 `image_dir` 복사 후 추론
- 출력 결과는 로컬에 쓰고 Drive 업로드 (`try/finally` 정리)
- 타이밍 출력: `캐시 Xs + 추론 Ys + 업로드 Zs = 합계s`

### 성능 영향 예측

| 병목 | 기존 | 개선 후 |
|------|------|---------|
| seed 이미지 Drive 읽기 | N × M회 | N × 1회 |
| Canny 계산 | N × M회 | N × 1회 |
| 배경 이미지 Drive 읽기 | N × M회 | 로컬 SSD (캐시) |
| GPU 추론 | N × M회 (지배적) | 동일 |

GPU 추론이 지배적이므로 전체 시간 단축은 10–20% 수준이나, Drive I/O 병목 환경에서는 더 클 수 있다.
